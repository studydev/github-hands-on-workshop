워크플로우 YAML 파일

.github/workflows/ 안에 있는 자동화 레시피

워크플로우는 저장소에 정의하는 자동화된 프로세스입니다. .github/workflows 디렉토리에 YAML 파일로 저장됩니다.

코드로 자동화를 정의하는 것 — 이것이 CI/CD의 핵심입니다.

기본 구조

name: Post welcome comment
on:
  pull_request:
    types: [opened]
permissions:
  pull-requests: write
jobs:
  welcome:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Hello!"

트리거 (이벤트)

워크플로우를 깨우는 신호

on: 키워드로 언제 워크플로우가 실행될지 정의합니다. 저장소에서 발생하는 다양한 이벤트에 반응할 수 있습니다.

자주 쓰는 트리거

  • push — 코드가 푸시될 때
  • pull_request — PR이 열리거나 업데이트될 때
  • issues — 이슈가 생성/수정될 때
  • schedule — cron 스케줄로 주기적 실행
  • workflow_dispatch — 수동 실행 버튼
이 실습에서는 pull_request: types: [opened]
→ 새 PR이 열릴 때 실행됩니다.

잡 (Jobs)

동일한 러너에서 함께 실행되는 단계들의 그룹

Job은 워크플로우 내에서 동일한 러너에서 함께 실행되는 step들의 그룹입니다. 각 job은 jobs 섹션 아래에 정의됩니다.

핵심 특징

  • 기본적으로 독립적이고 병렬로 실행
  • needs:로 의존 관계를 설정 가능
  • 빌드, 테스트, 배포 등 논리 단위로 구성
  • 매트릭스 전략으로 여러 변형 실행 가능
jobs:
  welcome:
    name: Post welcome comment
    runs-on: ubuntu-latest

스텝 (Steps)

Job 안에서 순차적으로 실행되는 개별 명령

Step은 job의 구성 요소로, 코드 체크아웃, 명령 실행, 오픈소스 Actions 사용 등의 작업을 자동화합니다.

두 가지 유형

  • run: — 셸 명령어 직접 실행
  • uses: — 미리 만들어진 Action 사용
steps:
  - run: gh pr comment "$PR_URL" \
        --body "Welcome!"
    env:
      GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
      PR_URL: ${{ github.event
                  .pull_request.html_url }}
Step은 순차 실행되지만, 각 Step은 독립 프로세스입니다.
변수 공유에는 inputs / outputs를 사용합니다.

러너 (Runners)

워크플로우를 실제로 실행하는 가상 머신

러너(Runner)는 워크플로우가 트리거되면 job을 실행하는 서버입니다. 각 러너는 한 번에 하나의 job을 실행합니다.

러너 유형

  • GitHub-hosted — GitHub가 관리하는 VM
    • ubuntu-latest (가장 많이 사용)
    • windows-latest
    • macos-latest
  • Self-hosted — 직접 관리하는 서버
퍼블릭 저장소는 GitHub-hosted 러너를 무료로 사용할 수 있습니다!

마켓플레이스 & 재사용

커뮤니티가 만든 수천 개의 Actions

GitHub Marketplace에는 커뮤니티가 만든 수천 개의 재사용 가능한 Action이 있습니다.

인기 Actions 예시

  • actions/checkout — 코드 체크아웃
  • actions/setup-node — Node.js 환경 설정
  • actions/cache — 의존성 캐싱
  • github/codeql-action — 보안 스캔
steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
    with:
      node-version: 20

실습의 비밀 — 자동 검증

뒤에서 자동으로 돌아가는 검증 시스템

이 실습은 GitHub Actions를 사용해 여러분의 진행을 자동으로 확인합니다.

자동 검증 흐름

  • 1단계: 워크플로우 파일 생성 → push 이벤트 → YAML 구조 확인
  • 2단계: Job 추가 → push 이벤트 → jobs 섹션 확인
  • 3단계: Step 추가 → push 이벤트 → steps 실행 로직 확인
  • 4단계: PR 생성 → pull_request → 워크플로우 실행 확인
  • 5단계: 병합 → pull_request closed → merged 확인
각 단계를 완료하면 Mona(봇)가 자동으로 다음 단계 안내를 남깁니다.

핵심 정리

Workflow

YAML로 정의 — .github/workflows/ 안에 저장

Trigger

on: 키워드로 이벤트 감지 — push, PR 등

Job

러너에서 실행되는 step 그룹 — 병렬 or 순차

Step

run: 또는 uses: — 개별 자동화 명령

Runner

워크플로우를 실행하는 VM — ubuntu-latest
