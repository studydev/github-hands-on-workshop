GitHub — 개발자의 플랫폼

Git은 소스 코드의 변경 이력을 추적하는 버전 관리 시스템입니다.

GitHub는 Git 저장소를 호스팅하고, 팀 협업을 위한 도구(Issue, PR, Actions 등)를 제공하는 클라우드 플랫폼입니다.

  • 전 세계 1억 8천만+ 개발자가 사용 (2025년 기준)
  • 4억 2천만+ 저장소가 호스팅된 세계 최대 플랫폼
  • Fortune 100 기업의 90%가 사용
  • 코드 호스팅 + 협업 + CI/CD + 보안 + AI

저장소 (Repository)

GitHub 안에 있는 프로젝트의 집

저장소는 프로젝트의 모든 파일과 폴더, 그리고 각 파일의 변경 기록(revision history)을 담는 공간입니다.

폴더처럼 보이지만, 내부에 모든 버전의 스냅샷이 기록되어 있습니다.

저장소에 포함되는 것들

  • 소스 코드 — 프로젝트의 실제 파일
  • README.md — 프로젝트 소개 문서
  • Issues — 버그 리포트, 기능 요청
  • Pull Requests — 코드 변경 제안
  • .github/ — 워크플로우, 템플릿 등 설정 파일

브랜치 (Branch)

저장소 안에서 갈라지는 병렬 작업 공간

브랜치는 저장소의 병렬 버전입니다. 기본 브랜치(main)의 복사본에서 독립적으로 작업한 뒤, 나중에 합칠 수 있습니다.

main ─────────── 안정 버전 (프로덕션)
  └── my-first-branch ─── 내 작업 공간 ← 여기!

왜 브랜치를 사용하나요?

  • 메인 코드를 망가뜨리지 않고 실험 가능
  • 여러 사람이 동시에 각자의 기능을 개발
  • 완성된 작업만 main에 합치기 (품질 관리)
실습에서는 my-first-branch라는 이름으로 브랜치를 만듭니다.

커밋 (Commit)

브랜치 안에서 변경을 기록하는 스냅샷

커밋은 파일의 추가, 수정, 삭제를 하나의 단위로 묶어 저장하는 것입니다. 프로젝트의 "타임머신" 역할을 합니다.

커밋의 구성 요소

  • 커밋 메시지 — 무엇을 왜 변경했는지 설명
  • 변경된 파일 목록 — diff (이전 vs 이후)
  • 작성자 & 시간 — 누가 언제 변경했는지
main ─────────── 안정 버전
  └── my-first-branch ── ● Add PROFILE.md ← 커밋!
실습에서는 PROFILE.md 파일을 만들고 첫 커밋을 합니다.

풀 리퀘스트 (Pull Request)

브랜치의 변경을 main에 합치자고 제안

풀 리퀘스트(PR)는 "내 브랜치의 변경을 main에 합쳐주세요"라고 제안하는 것입니다.

PR에서 할 수 있는 것

  • 변경된 코드를 나란히 비교 (diff)
  • 팀원에게 리뷰 요청
  • 코멘트로 토론
  • 수정 후 다시 커밋하면 PR에 자동 반영
실습에서는 my-first-branch → main 방향으로 PR을 만듭니다.

PR 생성 흐름

main ─────────────────── 안정 버전
  └── my-first-branch ── ● commit
          ↑ Pull Request ← 합쳐주세요!

병합 (Merge)

브랜치의 작업이 main에 최종 통합

병합은 PR의 변경 사항을 main 브랜치에 통합하는 최종 단계입니다.

main ──────── ● merge commit ──▶ main (업데이트!)
  └── my-first-branch ── ● commit ──┘ ← 병합 완료!

병합 후 해야 할 것

  • Merge pull request 클릭
  • Confirm merge 클릭
  • 사용한 브랜치 삭제 (Delete branch)
병합도 하나의 커밋입니다. 병합 커밋이 main에 기록됩니다.

GitHub Actions — 실습의 비밀

뒤에서 자동으로 돌아가는 검증 시스템

이 실습은 GitHub Actions를 사용해 여러분의 진행을 자동으로 확인합니다.

동작 원리

  • 1단계: 브랜치 생성 → create 이벤트 감지 → 브랜치 이름 확인
  • 2단계: 파일 커밋 → push 이벤트 감지 → PROFILE.md 존재 확인
  • 3단계: PR 열기 → pull_request 이벤트 감지 → PR 구조 확인
  • 4단계: 병합 → pull_request closed 감지 → merged 확인
각 단계를 완료하면 Mona(봇)가 자동으로 다음 단계 안내 코멘트를 남깁니다.

핵심 정리

Repository

프로젝트의 집 — 파일 + 히스토리

Branch

안전한 작업 공간 — main을 지키며 실험

Commit

변경의 스냅샷 — 메시지와 함께 기록

Pull Request

변경 제안 + 리뷰 + 토론의 장

Merge

최종 통합 — 작업 브랜치 → main
