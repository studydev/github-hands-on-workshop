머지 충돌 (Merge Conflict)

두 브랜치가 같은 줄을 수정하면 Git이 멈춥니다

머지 충돌은 두 개의 브랜치에서 같은 파일의 같은 부분을 수정했을 때 발생합니다. Git은 어느 쪽을 선택할지 사람에게 결정을 요청합니다.

main ── resume.md 추가 ── 스킬 동시 업데이트
  └── my-resume ── 스킬 업데이트 ── 충돌 발생!

충돌이 발생하는 조건

  • 같은 파일의 같은 줄을 두 브랜치에서 수정
  • 한쪽에서 파일을 삭제하고 다른 쪽에서 수정
  • 자동 머지가 불가능할 때만 발생
실습 1단계에서 my-resume → main 방향으로 PR을 만들면 충돌을 확인할 수 있습니다.

충돌 마커 (Conflict Markers)

Git이 남기는 특별한 표시를 읽어보자

충돌이 발생하면 Git은 파일에 3개의 마커를 삽입합니다:

<<<<<<< my-resume
- Contributed to open source projects
=======
- Built internal tools
>>>>>>> main
  • <<<<<<< — compare 브랜치(내 작업)의 시작
  • ======= — 두 버전의 경계선
  • >>>>>>> — base 브랜치(main)의 끝
마커와 원하지 않는 내용을 모두 삭제한 뒤 Mark as resolved를 클릭합니다.

충돌 해결 — 3가지 전략

어떤 버전을 남길지 결정하기

충돌을 해결하는 3가지 방법이 있습니다:

  • 1. Base 브랜치 수용 — main의 버전을 유지하고 내 변경 삭제
  • 2. Compare 브랜치 수용 — 내 버전을 유지하고 main 변경 삭제
  • 3. 수동 결합 — 두 버전을 합쳐서 새로운 내용 작성
main ───────────────────────── 안정 버전
  └── my-resume ── ● 충돌 해결 커밋 ── 작업 계속
충돌을 해결해도 PR이 자동으로 머지되지 않습니다. 역방향 머지 커밋이 compare 브랜치에 저장됩니다.

웹 편집기로 해결하기

GitHub에서 바로 충돌을 수정하는 과정

단계별 해결 과정

  • 1. PR 하단의 Resolve conflicts 버튼 클릭
  • 2. 충돌 마커가 강조된 섹션 찾기
  • 3. 불필요한 쪽 삭제 + 마커 3줄 모두 제거
  • 4. Mark as resolved → Commit merge
✓ 해결 후:
- Contributed to open source projects
실습 2단계에서 웹 편집기를 사용해 resume.md의 충돌을 직접 해결합니다.

PR 머지 (Merge)

충돌 해결 후 안전하게 병합

충돌을 해결하고 나면, PR의 Merge pull request 버튼이 다시 활성화됩니다.

my-resume ── ● ── 충돌 해결 ──┐
main ─────────────────── ● merge ← HEAD

머지 후 할 일

  • Merge pull request 클릭
  • Confirm merge 클릭
  • 사용한 브랜치 삭제 (선택 사항)
실습 3단계에서 Conversation 탭 하단의 Merge pull request를 클릭해 병합을 완료합니다.

GitHub Actions — 실습의 비밀

뒤에서 자동으로 돌아가는 검증 시스템

이 실습은 GitHub Actions를 사용해 여러분의 진행을 자동으로 확인합니다.

동작 원리

  • 1단계: PR 생성 → pull_request 이벤트 감지 → 브랜치 확인
  • 2단계: 충돌 해결 → push 이벤트 감지 → 마커 제거 확인
  • 3단계: PR 머지 → pull_request closed 감지 → merged 확인
각 단계를 완료하면 Mona(봇)가 자동으로 다음 단계 안내 코멘트를 남깁니다.
