교차 참조 (Cross-reference)

GitHub 안에서 점과 점을 연결하는 마법

다른 이슈나 풀 리퀘스트를 번호로 참조하면 하이퍼링크로 변환되고, 상대방에게도 교차 참조가 자동 생성됩니다.

참조 만드는 법

  • #5 → 같은 저장소의 이슈/PR #5로 링크
  • owner/repo#10 → 다른 저장소의 이슈 참조
  • 댓글에서 # 입력 시 자동완성 제안
  • 양방향 링크로 정보의 관계를 추적

중복 이슈 관리

여러 팀원이 같은 문제를 보고할 때

여러 팀원이 협업하다 보면 같은 문제에 대해 여러 이슈가 생길 수 있습니다. 교차 참조 기능으로 중복을 추적하고 적절하게 닫을 수 있습니다.

중복 이슈 처리 프로세스

  • 1. 기존 이슈 목록을 검색해 중복 확인
  • 2. 중복 이슈에 "Duplicate of #N" 댓글 작성
  • 3. 중복된 이슈를 Close as not planned로 닫기
  • 4. 원본 이슈에 자동으로 교차 참조 표시
이슈 #1 ──→ "Duplicate of #2" ──→ Close
실습 1단계에서 이슈 #1을 중복으로 표시하고 닫습니다.

Git Blame — 누가, 언제, 왜?

파일의 각 줄을 마지막으로 수정한 사람 추적

git blame은 파일의 각 줄을 마지막으로 수정한 리비전과 작성자를 보여주는 Git 기능입니다.

Git Blame 활용

  • 누가 커밋했는지 확인
  • 언제 변경이 이루어졌는지 추적
  • 왜 변경되었는지 커밋 메시지로 파악
  • GitHub에서 파일 → Blame 버튼 클릭

SHA (보안 해시 알고리즘)

  • 각 커밋의 고유 식별자 — 40자 16진수
  • 보통 처음 7자로 축약하여 사용
  • SHA를 이슈 댓글에 입력하면 커밋 링크로 변환
실습 2단계에서 docs/_sidebar.md의 Blame 뷰를 확인하고 커밋 SHA를 찾습니다.

커밋에서 맥락 찾기

코드 고고학 — 변경의 이야기를 추적하기

커밋은 단순히 코드 변경이 아닙니다. 각 커밋은 훨씬 더 많은 정보와 연결되어 있습니다:

  • 누가 커밋했는지 — 작성자 정보
  • 어떤 다른 변경이 포함되었는지 — diff 뷰
  • 언제 커밋이 이루어졌는지 — 타임라인
  • 커밋이 어떤 PR의 일부였는지 — PR 링크

커밋 → PR 추적

GitHub에서 커밋을 보면 해당 커밋이 생성된 풀 리퀘스트로의 링크를 확인할 수 있습니다. PR을 통해 왜 변경이 이루어졌는지, 누가 승인했는지, 어떤 테스트가 실행되었는지 알 수 있습니다.

히스토리 찾기는 누군가를 탓하는 것이 아니라 더 큰 그림을 보는 것입니다.

GitHub Actions — 실습의 비밀

뒤에서 자동으로 돌아가는 검증 시스템

이 실습은 GitHub Actions를 사용해 여러분의 진행을 자동으로 확인합니다.

동작 원리

  • 1단계: 이슈 닫기 → issues closed 이벤트 감지 → 중복 댓글 확인
  • 2단계: SHA 댓글 → issue_comment 감지 → 올바른 SHA인지 검증
  • 3단계: PR 머지 → pull_request closed 감지 → 사이드바 수정 확인
각 단계를 완료하면 Mona(봇)가 자동으로 다음 단계 안내 코멘트를 남깁니다.
