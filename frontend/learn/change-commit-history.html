민감한 데이터란?

저장소에 절대 있으면 안 되는 것들

민감한 콘텐츠란 저장소 히스토리에 체크인되어 사용자나 조직을 위험에 빠뜨릴 수 있는 모든 것입니다.

대표적인 민감 데이터

  • 비밀번호 — DB 접속 정보, 관리자 계정
  • API 키 / 액세스 토큰 — 클라우드, 외부 서비스 키
  • .env 파일 — 환경 변수에 담긴 시크릿
  • 개인 키 — SSH, TLS 인증서

노출 시 대응 모범 사례

  • 즉시 자격 증명 무효화 (토큰 재발급)
  • 모든 저장소 사본에서 완전 제거
  • 향후 노출 방지 조치 적용

git rm — 최신 커밋에서 파일 제거

첫 번째 단계: 현재 버전에서 삭제

가장 먼저 해야 할 일은 저장소의 현재 상태에서 파일을 삭제하는 것입니다.

git rm .env
git commit -m "remove .env file"
git push

git rm vs 수동 삭제

  • git rm — 워킹 디렉토리에서 삭제 + 스테이징까지 한 번에
  • rm + git add — 수동 삭제 후 따로 스테이징 필요
⚠️ git rm으로 삭제해도 Git 히스토리에는 여전히 남아 있습니다! 파일이 추가된 과거 커밋을 찾을 수 있습니다.

BFG Repo-Cleaner

Git 히스토리 전체에서 흔적 완전 삭제

BFG Repo-Cleaner는 저장소 히스토리를 검색하고 변경하는 전용 도구입니다. git filter-repo보다 간단합니다.

# 히스토리 확인 (최소 2개 커밋 보임)
git log --stat --all -- .env

# BFG로 모든 .env 참조 삭제
bfg --delete-files .env

# 히스토리 재확인 (비어있어야 함)
git log --stat --all -- .env

head 커밋이란?

  • HEAD는 현재 브랜치의 가장 최신 커밋을 가리킴
  • BFG는 HEAD 커밋은 보호하고 과거 히스토리만 정리
  • HEAD에서 먼저 파일을 삭제한 후 BFG 실행!

git push --force

재작성된 히스토리를 원격에 반영하기

BFG로 로컬 히스토리를 수정한 후, 원격 저장소에도 반영해야 합니다. 히스토리가 변경되었으므로 Force Push가 필요합니다.

# BFG 정리 후 로컬 리포 정리
git reflog expire --expire=now --all
git gc --prune=now --aggressive

# 강제 푸시
git push --force

⚠️ Force Push 주의사항

  • 원격 히스토리를 완전히 덮어씀 — 되돌리기 어려움
  • 협업자의 로컬 사본과 충돌 발생 가능
  • 팀원에게 기존 클론 삭제 & 새로 클론 요청 필수
  • 보호된 브랜치에서는 차단될 수 있음

.gitignore — 미래 커밋 방지

같은 실수를 두 번 하지 않기 위한 안전장치

.gitignore 파일에 패턴을 추가하면, Git이 해당 파일을 자동으로 무시합니다.

# .gitignore에 추가
.env
.env.local
*.pem
secrets/

적용 방법

.gitignore 편집 ──→ git add & commit ──→ git push
  • 로컬에 파일을 추가해도 GitHub에는 푸시되지 않음
  • 팀 전체에 적용하려면 저장소에 커밋
  • 글로벌 gitignore는 git config --global core.excludesFile

GitHub Actions — 실습의 비밀

뒤에서 자동으로 돌아가는 검증 시스템

이 실습은 GitHub Actions를 사용해 여러분의 진행을 자동으로 확인합니다.

동작 원리

  • 1단계: .env 제거 → push 이벤트 감지 → .env 삭제 확인
  • 2단계: BFG 실행 → push --force 감지 → 히스토리에서 .env 제거 확인
  • 3단계: .gitignore 추가 → push 이벤트 감지 → .env 패턴 확인
각 단계를 완료하면 Mona(봇)가 자동으로 다음 단계 안내 코멘트를 남깁니다.
